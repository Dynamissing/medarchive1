# Fix: Parsed Document Rows → Public Search Pipeline

## Root Cause

The document parsing pipeline works end-to-end: documents are parsed, rows are normalized into `PriceItemPayload`, matching runs against the `Service` table, and `PriceItemVersion` records are created. However, **parsed rows are never promoted into the `Service` table**. The public search endpoint (`GET /search`, `GET /services`) only queries the `Service` table — which is only populated by the separate `POST /admin/import/services` (directory import) flow. As a result, services that exist only in parsed documents are invisible to search.

Additionally, when the matching engine fails to find a match (score below threshold), `PriceItemVersion.service_id` is left `NULL`, so the join from `Service` → `PriceItemVersion` produces no results for those rows.

## Implementation Plan

### Step 1: Create `app/services/admin/parsed_service_publish.py`

New file implementing the publish bridge between parsed data and searchable `Service` records.

```python
# Key function:
def publish_parsed_services(
    db: Session,
    document: PriceDocument,
    rows: list[PriceItemPayload],
    price_versions: list[PriceItemVersion],
) -> dict:
    """
    After normalization+matching, ensure every unique normalized_service_name
    from parsed rows has a corresponding Service record and that all
    PriceItemVersion records are linked to it.
    
    Returns summary dict with counts of created/linked services.
    """
```

**Logic:**
1. Group `rows` by `normalized_service_name`
2. For each unique `normalized_service_name`:
   a. Compute `source_hash = sha256(normalized_service_name + ":" + str(document.id))` — deterministic per document
   b. Try to find an existing `Service` by `normalized_name` (any import batch) — reuse existing service if found
   c. If not found, create new `Service` with `source_type="parsed_document"`, `import_batch=str(document.id)`
   d. Generate `ServiceSynonym` records (reusing `generate_synonyms` from `service_directory_import.py`)
3. For each `PriceItemVersion` where `service_id is None`, find the matching `Service` by `normalized_service_name` and set `service_id`
4. Commit and return summary

**Deduplication strategy:**
- First check: `SELECT Service WHERE normalized_name = ?` (finds services from any source)
- If found: reuse, skip creation
- If not found: create with unique `(import_batch, source_hash)` constraint already on `Service` model
- On reprocess: existing Services persist, new ones only created if name didn't exist before
- PriceItemVersion links are updated each time (idempotent)

**Synonym generation:**
- Import `generate_synonyms` and `normalize_text` from `service_directory_import.py`
- Create synonyms with `source="parsed_document"` to distinguish from `"deterministic"` synonyms

### Step 2: Modify `app/workers/pipeline.py`

In `process_parsed_result()`, after the matching/recording loop (after line ~200), add a call to the publish function:

```python
# After the for-loop and before the return dict construction:
from app.services.admin.parsed_service_publish import publish_parsed_services

# Collect all versions created in this run
# (need to accumulate versions across iterations — currently overwritten each iteration)

publish_summary = publish_parsed_services(self.db, document, rows, all_versions)
summary.update(publish_summary)
```

**Specific change:** The current code overwrites `versions` each loop iteration. Need to accumulate them:
```python
# Before loop:
all_versions: list[PriceItemVersion] = []

# Inside loop:
versions = history_service.validate_and_record(row, ...)
all_versions.extend(versions)
recorded_price_items += len(versions)

# After loop:
from app.services.admin.parsed_service_publish import publish_parsed_services
publish_summary = publish_parsed_services(self.db, document, rows, all_versions)
```
