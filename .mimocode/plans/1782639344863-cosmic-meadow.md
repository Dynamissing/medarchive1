# Plan: Fix Stuck & Slow Documents

## Problem Summary
2 PDF documents stuck at 10% processing. Root cause: pipeline sets 10% before parsing, saves no intermediate progress. If parsing is slow or worker dies, document stays "processing" forever. No timeout, no parallelism config, no stale recovery.

## Key Files to Modify

### Backend
- `backend/app/workers/pipeline.py` — progress updates, timeout, stale detection
- `backend/app/workers/tasks.py` — task time limits, error handling
- `backend/app/workers/celery_app.py` — already has `prefetch_multiplier=1`
- `backend/app/services/parsers/pdf_text.py` — PyMuPDF fast path, pdfplumber limit
- `backend/app/services/parsers/pdf_ocr.py` — page/time limits for OCR
- `backend/app/services/document_processing.py` — pass progress callback to parsers
- `backend/app/db/models/archive.py` — add timing fields to PriceDocument
- `backend/app/api/routes/admin.py` — stale recovery endpoint, expanded detail
- `backend/app/core/constants.py` — new error codes
- `docker-compose.yml` — worker concurrency

### Frontend
- `frontend/features/documents/documents-page.tsx` — stuck filter, bulk reprocess, pagination
- `frontend/lib/api.ts` — new API functions

## Implementation Steps

### Step 1: Add Timing Fields to PriceDocument Model

**File**: `backend/app/db/models/archive.py`

Add to `PriceDocument`:
```python
processing_started_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True))
processing_finished_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True))
parser_stage: Mapped[str | None] = mapped_column(String(64))
```

Create Alembic migration.

### Step 2: Expand parsed_summary with Timing Data

**File**: `backend/app/workers/pipeline.py`

In `process_document()`:
- Record `processing_started_at` when starting
- Record `processing_finished_at` when done (success or fail)
- Store in `parsed_summary`: `duration_ms`, `parser_stage`, `processing_started_at`, `processing_finished_at`

In `process_parsed_result()`, accept and use a progress callback to update `parser_stage` at each phase:
- `"parsing"` → `"normalizing"` → `"matching"` → `"recording"`

### Step 3: Add Task-Level Timeout

**File**: `backend/app/workers/tasks.py`

```python
@celery_app.task(
    bind=True,
    max_retries=3,
    retry_backoff=True,
    soft_time_limit=180,  # 3 min soft limit
    time_limit=240,       # 4 min hard limit
)
def process_document_task(self, price_document_id: str, force: bool = False) -> dict:
```

Add `SoftTimeLimitExceeded` to the exception handling in pipeline.

### Step 4: Add Stale Document Recovery

**File**: `backend/app/workers/pipeline.py`

Add method:
```python
def recover_stale_documents(self, stale_threshold_minutes: int = 10) -> int:
    """Reset documents stuck in 'processing' for too long."""
```

Query: `PriceDocument.status == 'processing' AND PriceDocument.created_at < now - threshold`

**File**: `backend/app/api/routes/admin.py`

Add endpoint:
```python
@router.post("/admin/documents/recover-stale")
def recover_stale_documents(
    threshold_minutes: int = Query(10, ge=1, le=60),
    db: Session = Depends(get_db),
) -> dict:
```

### Step 5: Optimize PDF Text Parsing

**File**: `backend/app/services/parsers/pdf_text.py`

Current flow: PyMuPDF → if no text → pdfplumber fallback (unlimited)

Change to:
1. PyMuPDF fast path (already first)
2. pdfplumber fallback only if PyMuPDF text < 50 chars AND pdfplumber shows > 2x more text
3. Add page limit for pdfplumber: max 50 pages
4. Add warning if fallback is used: `"pdfplumber_fallback_used"`

No changes to PyMuPDF path — it's already the fast path.

### Step 6: Add OCR Page/Time Limits

**File**: `backend/app/services/parsers/pdf_ocr.py`

```python
OCR_MAX_PAGES = 30
OCR_PAGE_TIMEOUT_SECONDS = 15
```

In `ocr_pages()`:
- Limit to first `OCR_MAX_PAGES` pages
- Add per-page timing, abort if total exceeds threshold
- Return warning: `"ocr_page_limit_exceeded"` or `"ocr_timeout_exceeded"`

### Step 7: Add Intermediate Progress to Pipeline

**File**: `backend/app/workers/pipeline.py`

In `process_document()`, update progress_percent and parser_stage at key points:
```python
document.progress_percent = 10   # start
document.parser_stage = "parsing"
# ... after parsing ...
document.progress_percent = 40
document.parser_stage = "normalizing"
# ... after normalization ...
document.progress_percent = 60
document.parser_stage = "matching"
# ... after matching ...
document.progress_percent = 80
document.parser_stage = "recording"
# ... after recording ...
document.progress_percent = 100
document.parser_stage = "completed"
```

Flush after each stage update so progress is visible even if the next stage fails.

### Step 8: Update Admin Document Detail

**File**: `backend/app/api/routes/admin.py`

processing_started_at: datetime | None = None

### Step 10: Frontend — Stuck Filter & Bulk Reprocess

**File**: `frontend/features/documents/documents-page.tsx`

**File**: `frontend/lib/api.ts`

  return apiFetch<{ recovered: number }>(
